## File categories

The repository contains three distinct file types with different naming rules:

### 1. Root-level custom files (written by recipe `patch_files`)
These live at the RIME user data root (e.g. `rime_ice.custom.yaml`, `default.custom.yaml`). They use `.custom.yaml` naming because RIME auto-loads `{schema}.custom.yaml` for each schema. **Never create these manually** — the recipe system writes them.

### 2. Preset files (`others/presets/`)
Follow uniform `[environment]_[name].yaml` naming (e.g. `common_algebra.yaml`, `desktop_switches.yaml`). Three roles:

| Role | Pattern | Wraps in `patch:` | Example |
|------|---------|-------------------|---------|
| Pure data | clean root key (`algebra:`, `resets:`) | No | `common_algebra.yaml`, `common_switches.yaml` |
| De-patched intermediate | clean root key with `__include` | No | `common_rime_ice.yaml` |
| Entry point | `patch:` + flat keys | Yes | `desktop_rime_ice.yaml`, `mobile_t9.yaml` |

### 3. Recipe files (`others/recipes/`)
Plum deployment scripts. They specify `install_files` (what to copy to user dir) and `patch_files` (what `.custom.yaml` to generate).

## RIME modular config rules

These five rules govern the entire include architecture:

### Rule 1: Flat archive + namespace prefixing
All preset files reside flat in `others/presets/` — no subdirectories. File names use `[environment]_[name].yaml` (e.g. `common_`, `desktop_`, `mobile_`). The prefix provides alphabetical grouping equivalent to subdirectories with cleaner path resolution.

### Rule 2: Global root resolution + suffix omission
RIME resolves all `__include` paths from the **user data root** — physical relative paths like `../` are not supported. RIME auto-appends `.yaml` to include paths, so references must **omit the `.yaml` suffix** (e.g. `others/presets/common_algebra:/algebra`). For files named `xxx.custom.yaml`, the include must use `xxx.custom` (retaining the mid-segment).

### Rule 3: Flat key navigation forbidden
`__include` paths use `/` for hierarchical node traversal. A **flat key** like `speller/algebra:` (slash embedded in YAML key name) **cannot** be addressed directly (e.g. `:/patch/speller/algebra` fails). Instead, target the parent clean key (e.g. `:/patch`) — RIME recursively loads all children, safely including the flat key.

### Rule 4: De-patched intermediate nodes
Shared config library files (intermediate presets) must **not** wrap content in a `patch:` container. They output clean root keys like `algebra:` or `resets:`. This keeps include targets simple (`:/algebra`) and avoids flat-key navigation issues.

### Rule 5: Surgical flat-key patching at entry points
Entry point files that merge into RIME's root `.custom.yaml` **must** use flat keys like `speller/algebra:` inside their `patch:` block. This ensures the patcher performs **incremental merge** (preserving sibling keys like `alphabet`, `delimiter`) rather than **whole-tree replacement** (which would erase them).

## Modular include architecture

Configuration is deduplicated through a multi-level `__include` chain, with platform differentiation contained entirely in each platform's `switches` file (only `@0/reset` differs: `1` for desktop, `0` for mobile).

### Desktop include chain

```
<user_dir>/rime_ice.custom.yaml                   ← recipe patch_files writes one line
└── __include: others/presets/desktop_rime_ice:/patch
    │                           ← all paths resolve from RIME user data root
    ├── __include: others/presets/desktop_switches:/resets
    │   └── __include: others/presets/common_switches:/resets
    │       + switches/@0/reset: 1                                (desktop override)
    └── speller/algebra:                               ← flat key keeps patch semantics
        └── __include: others/presets/common_rime_ice:/algebra
            │                           ← de-patched intermediate: plain algebra: root
            ├── __include: others/presets/common_algebra:/algebra
            └── __append: (appended derive rules)
```

### Mobile include chain

```
<user_dir>/rime_ice.custom.yaml                   ← recipe patch_files writes one line
└── __include: others/presets/mobile_rime_ice:/patch
    ├── __include: others/presets/mobile_switches:/resets
    │   └── __include: others/presets/common_switches:/resets
    │       + switches/@0/reset: 0                                (mobile override)
    └── speller/algebra:
        └── __include: others/presets/common_rime_ice:/algebra
            ├── __include: others/presets/common_algebra:/algebra
            └── __append: (appended derive rules)
```

`desktop_rime_ice.yaml` and `mobile_rime_ice.yaml` are structurally identical — only the `switches` reference differs.

### T9 chain

```
<user_dir>/t9.custom.yaml                         ← recipe patch_files writes one line
└── __include: others/presets/mobile_t9:/patch
    ├── __include: others/presets/mobile_switches:/resets
    │   └── __include: others/presets/common_switches:/resets
    │       + switches/@0/reset: 0
    ├── engine/translators/@next:
    │     - table_translator@melt_eng
    └── speller/algebra:
        ├── __include: others/presets/common_algebra:/algebra  (skips rime_ice's appended derive rules)
        └── __append:
              - derive/^(.*)$/\U$1/            (uppercase derive)
              - xform/[ABC]/2/ through [WXYZ]/9/  (9-key mapping)
```

T9 includes `common_algebra` directly (not `common_rime_ice`) because it doesn't need the appended derive rules — it has its own uppercase derive and 9-key xform mappings.

### `__append` semantics

`__append` merges list items into the target node's existing list **without deduplication**. In `common_rime_ice.yaml`:
```yaml
algebra:
  __include: others/presets/common_algebra:/algebra
  __append:
    - derive/([aeio])ng$/$1mg/
    - derive/([gkh]|zh|sh)ua$/$1au/
    - derive/^([wghk])ai$/$1ia/
```
RIME first loads `common_algebra:/algebra`, then appends additional rules at the `speller/algebra:` position in the entry point.

## Recipe system

### Recipe types

| Recipe | Rx | Role |
|--------|-----|------|
| `desktop.recipe.yaml` | `others/recipes/desktop` | Patches 4 desktop custom files |
| `full.recipe.yaml` | `all` (rewritten by sync.py) | Installs framework + dictionaries |
| `grammar.recipe.yaml` | `others/recipes/grammar` | Downloads + patches LMDG grammar model |
| `mobile.recipe.yaml` | `others/recipes/mobile` | Patches 3 mobile custom files |

### Recipe `patch_files` mechanism

`patch_files` does **not** expand or inline includes at install time. It writes a single indirect `__include` line into the root-level custom file:

```yaml
# <user_dir>/rime_ice.custom.yaml  (written by recipe)
patch:
  __include: others/presets/desktop_rime_ice:/patch
```

At RIME load time, this delegates to `others/presets/desktop_rime_ice.yaml`, launching the full modular include chain. Plum merely writes the delegation line — all include resolution happens at runtime in librime.

**Important**: `__include` (and `__patch`) are librime-native YAML merge directives parsed by librime's C++ compiler at schema load time. Plum's minimal YAML parser does **not** support `__include` — never use it directly in recipe YAML (it will crash Plum). It is safe inside preset files.

### Grammar recipe (parameterized)

`grammar.recipe.yaml` uses `${schema}` argument substitution:
```bash
bash rime-install others/recipes/grammar.recipe.yaml schema=t9
```
The recipe writes to `${schema:-rime_ice}.custom.yaml`, defaulting to `rime_ice` if no schema is specified. It patches `grammar/` settings directly with flat keys (no `__include` chain).

### Deployment requirements

`full.recipe.yaml` must include `- others/presets/*.*` in `install_files` so all preset files are present in the RIME user directory at runtime. The platform recipes (`desktop.recipe.yaml`, `mobile.recipe.yaml`) need no `install_files` — they only write patch delegations.

## sync.py pipeline

`others/scripts/sync.py` orchestrates the CI build. Steps:

1. **Download**: Fetches `iDvel/rime-ice` (main) and `gaboolic/rime-frost` (master) as zip archives
2. **Extract**: Unpacks into `temp_build/`
3. **Clean root**: Wipes all root files **except** those whitelisted by `.gitignore` `!/` patterns plus `.git/` and `temp_build/`
4. **Copy ice framework**: Copies `ICE_KEEP_FILES` (schemas, lua, opencc, en_dicts, default.yaml, configs) to root
5. **Copy frost dictionaries**: Copies `cn_dicts/` and `cn_dicts_cell/` to root
6. **Rewrite schema_list**: Replaces `default.yaml`'s `schema_list` with only `rime_ice` and `t9`
7. **Dictionary swap**: Reads `others/presets/rime_frost.dict.yaml`, rewrites `name: rime_frost` → `name: rime_ice`, writes to root `rime_ice.dict.yaml`
8. **Rewrite recipe**: Reads `others/recipes/full.recipe.yaml`, rewrites its `recipe` block (`Rx: all`), writes to root `recipe.yaml`
9. **Normalize line endings**: Strips `\r` from `.lua`, `.yaml`, `.txt`, `.dict`, `.schema` files

### ICE_KEEP_FILES

```
custom_phrase.txt
default.yaml
en_dicts/
lua/
melt_eng.dict.yaml
melt_eng.schema.yaml
opencc/
radical_pinyin.dict.yaml
radical_pinyin.schema.yaml
rime_ice.schema.yaml

```

### `.gitignore` dynamic whitelist

The base `.gitignore` starts with `/*` (ignore everything), then uses `!/` patterns to whitelist specific paths.

`get_clean_whitelist()` reads `.gitignore` and extracts `!/` prefixed entries (`.gitignore`, `.github/`, `AGENTS.md`, `LICENSE`, `others/`, `README.md`). These — plus `.git/` and `temp_build/` — are **protected from deletion** during `clean_root_directory()`. Everything else at root is wiped before each CI run.

## CI behavior

### Build workflow (`.github/workflows/build.yml`)

`workflow_dispatch` only — two sequential jobs:

1. **Build**: Runs `sync.py` to merge upstreams, commits all generated root files, force-pushes to `main`.
2. **Releases**: Checks out `main`, uses plum to install 4 profile combinations via `rime-install` (with a `git ls-remote` retry loop to ensure the Build push has propagated to GitHub), compresses each to `.7z`, uploads to a rolling `latest` release.

### Release artifacts

| Archive | Contents |
|---------|----------|
| `desktop.7z` | Desktop (全拼, no grammar) |
| `desktop-grammar.7z` | Desktop (全拼 + LMDG grammar) |
| `mobile.7z` | Mobile (全拼 + T9, no grammar) |
| `mobile-grammar.7z` | Mobile (全拼 + T9 + LMDG grammar) |

Archives are built with plum's `rime-install` — identical to a user running plum locally. Unzip to the RIME user directory and redeploy.

- **No-op**: If upstreams are unchanged, Build produces no commit. Releases still rebuilds and uploads.
- **Local equivalent**: `python others/scripts/sync.py` — will purge root files, only run when you intend a full rebuild.

## Dictionary swap
The script reads `others/presets/rime_frost.dict.yaml` (or falls back to frost's default), rewrites `name: rime_frost` → `name: rime_ice` in the YAML, and writes it to `rime_ice.dict.yaml` at the root. This makes the frost dictionary appear as rime-ice's main dictionary to the schema.

## Adding root files
**Adding a root file requires updating two places:**
1. `.gitignore` `!/` whitelist — `get_clean_whitelist()` reads this file dynamically
2. `others/recipes/full.recipe.yaml` `install_files` — ensures the file is present in the RIME user directory at deployment time

## Style rule
Comments must be on their own line — never inline with code.

Chinese prose in `README.md` uses half-width punctuation only:
colons `:`, commas `,`, semicolons `;`, parentheses `()`, periods `.`.
No Chinese full-width punctuation (`：。、；（）`) or terminal sentence periods.

## Chosen schemas
Only two schemas survive in `default.yaml`: `rime_ice` and `t9`. `sync.py` enforces this by regex-replacing the upstream `schema_list` block, removing all other schemes.
